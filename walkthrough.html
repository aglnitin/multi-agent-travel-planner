Multi-Agent Travel Planner

A full walkthrough using the example: "Fly from Delhi to Bangkok, July 10–17 2026, 2 adults, mid-range"

❓ Why are LangChain AND LangGraph used together?

This is the most common source of confusion. Here is the one-sentence answer:

LangGraph is the workflow engine. LangChain is the toolbox it uses inside each step. They are not alternatives — LangGraph is built on top of LangChain.

Think of it this way:

🔮
LangGraph — The Workflow Engine
Orchestrates WHEN each piece runs. Defines nodes (steps), edges (transitions), conditional routing, checkpointing, and memory across turns.
StateGraph add_node / add_edge add_conditional_edges MemorySaver START / END
🔷
LangChain — The AI Toolkit
Provides the individual AI building blocks: LLM wrappers, tool decorators, prompt templates, structured output, and the ReAct agent loop.
ChatOpenAI @tool with_structured_output() create_react_agent SystemMessage / HumanMessage
🐍
Python — Glue + External APIs
asyncio.gather for parallel execution. requests for SerpAPI / Tavily HTTP calls. Telegram bot handler. Pydantic for schema validation.
asyncio.gather requests.get pydantic BaseModel python-telegram-bot
🔷 LangChain — "what can it do?"
  • Call OpenAI / Anthropic / any LLM
  • Decorate a Python function as an LLM tool
  • Force the LLM to return a Pydantic schema
  • Run a single ReAct agent loop (Reason → Act → Observe)
  • Format system + human messages
🔮 LangGraph — "in what order?"
  • Define a graph of steps (nodes)
  • Route between steps conditionally
  • Pass typed state between steps
  • Remember conversations across turns (MemorySaver)
  • Checkpoint so a crash can be resumed
Analogy: LangChain is like React hooks (useState, useFetch, useQuery) — individual capabilities. LangGraph is like Next.js — the routing and page lifecycle that decides when each hook runs and in what order.

🔍 Step-by-Step Trace

Following the message "I want to fly from Delhi to Bangkok on July 10 2026 and return July 17 2026, 2 adults, mid-range budget" through every line of code.

0
User sends message → Telegram Bot receives it src/bot.py → handle_message() python-telegram-bot

The Telegram bot gets a webhook/polling event. It extracts the text and chat ID, then hands off to LangGraph.

src/bot.py
async def handle_message(update, context): chat_id = update.message.chat_id # e.g. 123456789 user_text = update.message.text # "I want to fly from Delhi..." initial_state = { "messages": [], "session_id": str(chat_id), # "123456789" "user_input": user_text, "trip_request": None, # filled by coordinator "orchestrator_output": None, # filled by orchestrator "final_output": None, # filled by validator } config = {"configurable": {"thread_id": str(chat_id)}} # thread_id = chat_id → per-user memory result = await travel_graph.ainvoke(initial_state, config=config) await update.message.reply_text(result["final_output"])
LangGraph role here: travel_graph.ainvoke() is the LangGraph graph compiled in src/graph.py. It accepts the state dict and runs it through all nodes automatically. You don't manually call coordinator → orchestrator → validator; LangGraph does that routing.
1
coordinator_node — Parse the user message src/agents/coordinator.py LangChain

The coordinator calls OpenAI and forces it to return a Pydantic-validated JSON object — no freeform text. This is LangChain's structured output feature.

src/agents/coordinator.py
async def coordinator_node(state: TravelState) -> dict: # 1. Create the LLM (this is LangChain) llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) # 2. Attach a Pydantic schema — LLM MUST return JSON matching this structured_llm = llm.with_structured_output(TripRequestSchema, method="json_mode") # 3. Build the messages array (LangChain message types) messages = [ SystemMessage(content="You are the Trip Request Coordinator..."), HumanMessage(content=state["user_input"]), # "I want to fly from Delhi..." ] # 4. Call OpenAI → get back a typed Python object, not raw text result: TripRequestSchema = await structured_llm.ainvoke(messages) # 5. Return updates to LangGraph state (LangGraph merges this into state) return {"trip_request": result.model_dump()}

What the LLM produces (structured output):

TripRequestSchema (Pydantic model output)
{ "origin": "Delhi", "destination": "Bangkok", "departure_date": "2026-07-10", "return_date": "2026-07-17", "number_of_travelers": 2, "travel_style": "mid-range", "error": false, ← "false" means all fields present "error_msg": "" }
Why structured output? Without it, the LLM might say "Sure! Here are the details..." — free text you'd need to regex-parse. with_structured_output(TripRequestSchema) instructs OpenAI to use its function-calling API internally and return only the JSON object. Pydantic validates it. If the LLM returns garbage, Pydantic raises an error before bad data propagates.
2
LangGraph routes: error? → END or orchestrate src/graph.py + src/agents/coordinator.py LangGraph

This is pure LangGraph — no LLM involved. A Python function reads the state and returns a string key that tells LangGraph which node to run next.

src/agents/coordinator.py — routing function
def route_after_coordinator(state: TravelState) -> str: trip = state.get("trip_request") or {} if trip.get("error", True): return "error_end" # → error node → sends error_msg → END return "orchestrate" # → runs flights + hotels + activities
src/graph.py — where the routing is wired up
graph.add_conditional_edges( "coordinator", # from this node... route_after_coordinator, # ...call this function to decide... { "error_end": "error_end", # ...route to this node if "error_end" "orchestrate": "orchestrate", # ...route to this node if "orchestrate" }, )
Our example has error=false, so LangGraph routes to orchestrate_node. If the user had said "Hi" (a greeting), error=true → LangGraph skips the API calls entirely and sends the error_msg directly. This is the same as the n8n "If" node.
3
orchestrate_node — 3 sub-agents run IN PARALLEL src/agents/orchestrator.py LangChain (ReAct agents) SerpAPI + Tavily

Three ReAct agents are created (LangChain) and run simultaneously with asyncio.gather (Python). Each agent reasons about what tool to call, calls it, reads the result, and returns formatted text.

src/agents/orchestrator.py — parallel execution
async def orchestrate_node(state: TravelState) -> dict: trip = state["trip_request"] trip_prompt = json.dumps(trip) # pass the whole trip JSON to each agent # asyncio.gather → ALL THREE start at the same time (parallel!) flights_result, hotels_result, activities_result = await asyncio.gather( _run_agent("gpt-4o-mini", FLIGHTS_SYSTEM, [search_flights], trip_prompt), _run_agent("gpt-4o-mini", HOTELS_SYSTEM, [search_hotels], trip_prompt), _run_agent("gpt-4o-mini", ACTIVITIES_SYSTEM, [search_local_sightseeing, search_tavily], trip_prompt), return_exceptions=True # if one fails, others still complete ) async def _run_agent(model, system_prompt, tools, trip_prompt) -> str: llm = ChatOpenAI(model=model, temperature=0) agent = create_react_agent(model=llm, tools=tools, prompt=system_prompt) result = await agent.ainvoke({"messages": [HumanMessage(content=trip_prompt)]}) return result["messages"][-1].content # last message = agent's final answer

What each sub-agent does (all 3 running at the same time):

✈️ Flights Agent
Tool: search_flights
Calls: SerpAPI Google Flights
Params: DEL → BKK, 2026-07-10, 2026-07-17, adults=2
Returns: 6 flight options with prices in INR
🏨 Hotels Agent
Tool: search_hotels
Calls: SerpAPI Google Hotels
Params: Bangkok, check-in 07-10, check-out 07-17
Returns: 7 hotel options with prices in INR
🗺 Activities Agent
Tools: search_local_sightseeing + search_tavily
Calls: SerpAPI Local AND Tavily web search
Returns: 10-12 activity options grouped by area
Timing benefit: Sequential would take ~15s (5s × 3). With asyncio.gather, all 3 API calls overlap → total ~5s. This is exactly what n8n's parallel "agentTool" nodes did.
4
Inside each sub-agent: the ReAct loop LangChain — create_react_agent ReAct Pattern

create_react_agent implements Reason → Act → Observe → Repeat. The LLM doesn't just call a tool blindly — it first reasons about what to call, then reads the result and decides if it needs to call again or answer.

Flights Agent — internal ReAct turns (what the LLM actually "thinks")
Turn 1 — REASON (LLM thinks): "I need to find flights from Delhi to Bangkok. Delhi's airport code is DEL, Bangkok's is BKK. Dates: 2026-07-10 outbound, 2026-07-17 return, 2 adults." Turn 1 — ACT (LLM calls the tool): search_flights( departure_id = "DEL", arrival_id = "BKK", outbound_date= "2026-07-10", return_date = "2026-07-17", adults = 2 ) Turn 1 — OBSERVE (tool returns to LLM): {"flights": [{"airline":"IndiGo","price_INR":65704,...}, ...]} Turn 2 — REASON + FINAL ANSWER (LLM formats the response): "I have the data. Let me format the top options..." → "1. IndiGo DEL→BKK, 15:45-21:45, Non-stop, ₹65,704..."
Why ReAct instead of just calling the tool directly? Because the agent needs to determine airport codes from city names (Delhi → DEL, Bangkok → BKK). The LLM uses its world knowledge to do this mapping before calling the tool. A plain function call couldn't do that without a hard-coded lookup table.
5
Inside the tool: HTTP request to SerpAPI src/tools/flights.py SerpAPI @tool decorator

The @tool decorator is a LangChain feature that turns a plain Python function into something the LLM can call. It reads the docstring + type hints to build the JSON schema that OpenAI uses for function calling.

src/tools/flights.py
@tool ← LangChain decorator — registers this as an LLM-callable tool def search_flights(departure_id: str, arrival_id: str, outbound_date: str, return_date: str, adults: int = 1) -> str: """Search round-trip flights via SerpAPI Google Flights.""" # ↑ docstring becomes the tool description the LLM reads # ↑ type hints become the JSON schema the LLM uses to call it params = { "engine": "google_flights", "departure_id": departure_id, # "DEL" "arrival_id": arrival_id, # "BKK" "outbound_date": outbound_date, # "2026-07-10" "return_date": return_date, # "2026-07-17" "adults": adults, "currency": "INR", "api_key": SERPAPI_API_KEY, } response = requests.get("https://serpapi.com/search", params=params, timeout=30) data = response.json() flights = [] for group in data.get("best_flights", []): legs = group.get("flights", []) flights.append({ "airline": legs[0].get("airline"), "price_INR": group.get("price"), "stops": len(legs) - 1, }) return json.dumps({"flights": flights[:6]})
6
validate_node — Curate and format the final itinerary src/agents/validator.py LangChain LangGraph node

The Validator receives the raw combined output (~3,000 tokens) from all 3 agents and produces the final formatted itinerary. It's a simple LLM call — no tools, no ReAct loop — just a strong system prompt.

src/agents/validator.py
async def validate_node(state: TravelState) -> dict: llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.3) # Combine everything into one context context = f"TRIP REQUEST:\n{state['trip_request']}\n\n" f"RAW AGENT OUTPUTS:\n{state['orchestrator_output']}" messages = [ SystemMessage(content=VALIDATOR_SYSTEM), # "Select top 2-3 flights..." HumanMessage(content=context), ] response = await llm.ainvoke(messages) return {"final_output": response.content} # the formatted itinerary

7
MemorySaver — Conversation persists for follow-up messages src/graph.py LangGraph

LangGraph's MemorySaver checkpoints the full state after every node. The next message from the same user picks up where it left off — this is how follow-ups work without re-explaining context.

src/graph.py — memory wired into the graph
# One line — that's it memory = MemorySaver() graph = graph.compile(checkpointer=memory) # Every invocation with the same thread_id shares memory config = {"configurable": {"thread_id": "123456789"}} # Turn 1: "I want to fly Delhi to Bangkok July 10-17" await graph.ainvoke(state1, config) # → full itinerary # Turn 2: "actually make it 3 people" ← coordinator sees previous context await graph.ainvoke(state2, config) # → knows it's Delhi→Bangkok, updates travelers
n8n equivalent: This is exactly the memoryBufferWindow node keyed by message.chat.id that was connected to the CoOrdinator agent. In LangGraph, MemorySaver + thread_id replaces it with zero additional nodes.

⚙️ The ReAct Loop — Visualised

create_react_agent is a LangChain / LangGraph prebuilt agent that implements the ReAct pattern. Here's the Flights Agent's full internal loop:

REASON
LLM reads system prompt + trip JSON. Thinks: "DEL → BKK, 2 adults, dates are 2026-07-10 to 2026-07-17."
ACT
search_flights("DEL", "BKK", "2026-07-10", "2026-07-17", 2)
OBSERVE
{"flights": [{"airline":"IndiGo","price_INR":65704,...},...], "count":6}
REASON
Has enough data. No need to call tool again. Ready to format final answer.
ANSWER
"1. IndiGo DEL→BKK, 15:45–21:45, Non-stop, ₹65,704\n2. Air India..."
Key insight: The LLM decides when it has enough information. If SerpAPI returned zero results, it could call the tool again with different parameters — or explain that no flights were found. This "reason before acting" is what separates a ReAct agent from a plain function call.
